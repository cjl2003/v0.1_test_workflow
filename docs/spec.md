## 一、题目名称
**基于全流程国产数字 EDA 系统的 MAC16 芯片设计**

## 二、参赛组别
A 组、B 组

## 三、赛题介绍
累积乘加运算（Multiply Accumulate，MAC）是各类 CPU、DSP、AI 芯片中常用的基础运算模块，用于实现乘法与累加的合并操作。

本题要求设计一款基于 **55nm 工艺** 的 **MAC16 数字芯片**：
- 输入：16 bit 串行无符号数据
- 输出：24 bit 串行无符号数据
- 功能：完成乘加累积与模式相关的输出处理

题目配套提供了总体功能说明，可参考如下框架理解模块结构。

## 四、MAC16 框架图

```text
                 +-----------------------------------+
 mode ---------->|                                   |
 inA ----------->|                                   |
 inB ----------->|               MAC16               |----> sum_out
 clk ----------->|                                   |----> carry
 rst_n --------->|      +-----------+                |----> out_ready
 in_ready ------>|      |  Multiply |----+           |
                 |      +-----------+    |           |
                 |                       v           |
                 |                  +---------+      |
                 |                  |  Accum  |      |
                 |                  +---------+      |
                 |                                   |
                 +-----------------------------------+
```

## 五、接口定义（Pin 信息）

1. **inA、inB**  
   输入 16 bit 串行信号，输入速率为 **1Gbps**，**MSB 先入**，由 `clk` 上升沿驱动采样。

2. **in_ready**  
   输入同步信号。当 `inA` 和 `inB` 正在输入时为 `1`，无数据输入时为 `0`。

3. **sum_out**  
   输出 24 bit 串行信号，输出速率为 **1Gbps**，**MSB 先出**，由 `clk` 上升沿驱动输出；无数据输出时窗口保持 `0` 电平。

4. **carry**  
   输出进位/溢出指示信号。默认值为 `0`；当发生相应进位/溢出并需要指示时置 `1`。  
   **补充约束：`carry` 置 `1` 后，可以被 `rst_n` 或 `mode` 切换清零。**

5. **out_ready**  
   输出同步信号。当 `sum_out` 正在输出时置 `1`，无数据输出时为 `0`。

6. **clk**  
   1GHz 主时钟，上升沿有效。在非复位状态下，模块在 `clk` 上升沿采样 `inA`/`inB` 串行数据，在模块内部完成乘法和累积运算，并将结果更新至 `sum_out` 串行输出。

7. **rst_n**  
   低有效复位信号。`rst_n=0` 时，整个电路不工作，并清空所有内部寄存器和所有输出。

8. **mode**  
   模式选择信号。  
   - `mode=0`：输出当前累积和与上一结果相关的累积结果；
   - `mode=1`：进行另一种全加式累积求和输出。  

   **补充约束（重要）：**
   - `mode` 切换时，不能直接丢弃切换前已经输入的数据；
   - **在 `mode` 切换之前已经输入的数据，必须按切换前的旧 `mode` 继续输出，直到该批数据输出完成；**
   - 该批旧模式数据输出完成后，再将相关寄存器数据清零；
   - 清零完成后，后续若继续输入数据，则必须按**切换后的新 `mode`** 进行处理和输出。

9. **数据属性补充说明**  
   **输入数据与输出数据均按无符号数据处理。**

## 六、测试组合说明

| 场景 | inA（十进制） | inB（十进制） | mode |
|---|---|---|---|
| 组合 1 | 2, 8, 14, 116, 1546, 20698 | 6, 30, 71, 828, 1152, 728 | 0 |
| 组合 2 | 2, 8, 14, 116, 1546, 20698 | 6, 30, 71, 828, 1152, 728 | 1 |
| 组合 3 | 2, 8, 14, 116, 1546, 20698 | 6, 30, 71, 828, 1152, 728 | 0→1（在 `14` 与 `71` 这一组输入完成后切换） |

## 七、测试时序要求
每个测试电路应满足如下时序约束：

1. `clk` 始终输入 **1GHz** 时钟信号；
2. 仿真开始后先令 `rst_n=0`，持续 **2 个 `clk` 周期** 后再拉高，并在之后保持有效；
3. 随后通过 `inA`、`inB` 串行输入 6 组数据；
4. `in_ready` 用于输入同步，每组输入之间允许有间隙，也可以连续输入；
5. 每组数据之间的间隔**不超过 5 个 `clk` 周期**；
6. 输出信号 `sum_out`、`carry`、`out_ready` 应覆盖完整输出过程；
7. `testbench` 需完成结果比较：
   - 任一组数据不符合预期，则打印 `Simulation Failed`；
   - 全部 6 组数据均正确，则打印 `Simulation Passed`。

## 八、补充题目约束条件（澄清修订版）
为避免对题意产生歧义，补充约束条件如下。

### 8.1 mode 切换行为（保留原意并明确实现口径）
1. `mode` 只会在一组数据输入完成后才会切换，`mode`切换时，相关寄存器最终需要清零；
2. 但在 `mode` 切换之前已经输入的数据，**必须仍按旧 `mode` 处理并完整输出**；
3. 旧模式下待输出数据全部输出完成后，再清零相关寄存器；
4. 清零完成后，后续新输入数据必须按**切换后的新 `mode`** 处理和输出。

### 8.1.1 mode 切换期间的输入接收规则（新增澄清）
为统一 RTL 与 testbench 的实现口径，现对 `mode` 切换期间的输入行为作如下明确：

1. **不得依赖长时间暂停输入来完成 `mode` 切换。**  
   由于题目规定每组输入之间的间隔为 **0~5 个 `clk` 周期**，因此设计不应假设外部会在 `mode` 切换后长时间停止送数。

2. **旧模式待输出结果尚未完全输出完毕时，允许继续接收后续新输入数据。**  
   但这些新输入数据在此阶段**仅可被接收并缓存**，不得提前参与新模式下的输出结果生成。

3. **切换前已经输入完成的数据，其输出优先级高于切换后新到达的数据。**  
   即：凡是在 `mode` 切换动作发生前已经被模块接收的数据，必须全部按照**旧 `mode`** 完整处理并输出，不得被切换后的新数据插队、覆盖或打断。

4. **切换后新接收的数据，必须等到旧模式待输出结果全部输出完成、且相关寄存器清零之后，方可按新 `mode`` 开始处理和输出。**

5. **允许“先接收、后处理”，不允许“先接收、先按新模式输出”。**  
   换言之，`mode` 切换后新到达的数据可以先进入输入缓冲或寄存器队列，但在旧模式结果尚未完全输出前，这些数据不得参与新模式计算结果的输出。

6. **testbench 判定口径应与上述行为保持一致。**  
   即：`mode` 切换后的新输入是否已经被模块接收，不作为旧模式结果是否完成输出的判定依据；判定依据应以旧模式待输出结果是否已完整输出为准。

### 8.1.2 组合 3（0→1 切换）的判定口径（新增澄清）
针对测试组合 3：

1. `mode` 从 `0` 切换到 `1` 的动作发生点，为 **`14` 与 `71` 这一组输入完成后**；
2. 在该切换点之前已经完成输入的 3 组数据，必须全部按 **旧 `mode=0`** 的规则处理并输出；
3. 从第 4 组开始到达的新输入数据，可在旧模式结果输出期间被接收，但不得提前按 `mode=1` 输出；
4. 必须在前 3 组旧模式结果全部输出完成、相关寄存器清零后，后续数据才可按 **新 `mode=1`** 规则处理和输出。

### 8.2 carry 清零条件
- `carry` 置 `1` 后，可以通过以下两种方式清零：
  1. `rst_n` 复位；
  2. `mode` 切换。

### 8.3 有符号/无符号属性
- `inA`、`inB`、`sum_out` 所涉及的数据，均按**无符号数**理解和处理。

### 8.4 输入采样结束到开始输出的延迟约束
1. 输入采样结束到开始输出的延迟，**没有固定 `clk` 周期限制**；
2. 允许参赛者为满足时序目标，对内部流水级数与流水切分位置进行调整；
3. 若内部流水变化导致输入到输出的延迟变化，则必须同步更新 `testbench`、说明文档及相关注释中的 latency 预期；
4. 上述 latency 可变这一事实，**不影响** 8.1.1 与 8.1.2 所规定的 `mode` 切换处理顺序和输出先后关系。
