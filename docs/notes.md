# MAC16 初学者说明笔记

这份笔记不是代码设计文档，而是把题目要求翻成更容易理解的中文说明。目标很简单：先把规则看懂，再去写 RTL 和 testbench，就不容易把 `mode`、`carry`、串行位序这些关键点写错。

## 1. 题目在做什么（白话版）

这个题目要做一个 `MAC16` 模块。它每次会从 `inA` 和 `inB` 串行收到一组 16 bit 无符号数，先做乘法，再按当前 `mode` 的规则生成结果，最后从 `sum_out` 串行送出 24 bit 无符号结果。

可以先把几个接口理解清楚：

- `inA`、`inB`：16 bit 串行输入，都是无符号数，`MSB first`，也就是最高位先进入。
- `in_ready`：输入有效标志。为 `1` 时，表示当前这几个时钟周期正在送输入位。
- `sum_out`：24 bit 串行输出，也是无符号数，也是 `MSB first`。
- `out_ready`：输出有效标志。为 `1` 时，表示当前这几个时钟周期正在输出结果位。
- `clk`：主时钟，上升沿采样和更新。
- `rst_n`：低有效复位。拉低后，内部寄存器和输出都要清空。
- `mode`：决定“乘完以后怎么组合结果”。
- `carry`：溢出标志。它不是打一拍就自动清零，而是会保持，只有 `rst_n` 或 `mode` 切换时才能清零。

题目至少要求覆盖 3 种情况：

1. 一直工作在 `mode 0`
2. 一直工作在 `mode 1`
3. 先工作在 `mode 0`，再在指定位置切换到 `mode 1`

题目里的 6 组输入数据是：

- `(2,6)`
- `(8,30)`
- `(14,71)`
- `(116,828)`
- `(1546,1152)`
- `(20698,728)`

先把每组乘积算出来，后面理解两种模式就会容易很多：

- `P[1] = 2 × 6 = 12`
- `P[2] = 8 × 30 = 240`
- `P[3] = 14 × 71 = 994`
- `P[4] = 116 × 828 = 96048`
- `P[5] = 1546 × 1152 = 1780992`
- `P[6] = 20698 × 728 = 15068144`

也就是：

`P = 12, 240, 994, 96048, 1780992, 15068144`

## 2. mode 0、mode 1、mode 切换到底是什么意思

这一节最重要。后面 RTL 和 testbench 写对写错，基本都卡在这里。

### 2.1 mode 0 是什么

`mode 0` 可以直接理解成：

“第 1 组先直接输出自己；从第 2 组开始，输出 = 当前乘积 + 上一组乘积。”

写成公式就是：

- `raw[1] = P[1]`
- `raw[n] = P[n] + P[n-1]`，其中 `n >= 2`

把上面的乘积表代进去：

- `raw[1] = 12`
- `raw[2] = 240 + 12 = 252`
- `raw[3] = 994 + 240 = 1234`
- `raw[4] = 96048 + 994 = 97042`
- `raw[5] = 1780992 + 96048 = 1877040`
- `raw[6] = 15068144 + 1780992 = 16849136`

所以 `mode 0` 的原始结果是：

`raw = 12, 252, 1234, 97042, 1877040, 16849136`

但是输出口 `sum_out` 只有 24 bit，所以真正送出去的是 `raw[23:0]`：

- 前 5 组都没超过 24 bit 范围，所以不变。
- 第 6 组 `16849136` 超过了 `2^24 - 1 = 16777215`，因此低 24 bit 会变成 `71920`，并且 `carry = 1`。

所以 golden vectors 给出的结果是：

- `sum_out[23:0] = 12, 252, 1234, 97042, 1877040, 71920`
- `carry = 0, 0, 0, 0, 0, 1`

一句话记忆：

`mode 0` 不是“全累加”，而是“当前乘积 + 上一组乘积”。

### 2.2 mode 1 是什么

`mode 1` 可以理解成：

“从当前模式开始，一直把所有乘积往后累加。”

写成公式就是：

- `raw[1] = P[1]`
- `raw[n] = raw[n-1] + P[n]`，其中 `n >= 2`

把乘积代进去：

- `raw[1] = 12`
- `raw[2] = 12 + 240 = 252`
- `raw[3] = 252 + 994 = 1246`
- `raw[4] = 1246 + 96048 = 97294`
- `raw[5] = 97294 + 1780992 = 1878286`
- `raw[6] = 1878286 + 15068144 = 16946430`

所以 `mode 1` 的原始结果是：

`raw = 12, 252, 1246, 97294, 1878286, 16946430`

同样因为输出口只有 24 bit，第 6 组要截成低 24 bit：

- `16946430` 的低 24 bit 是 `169214`
- 这一组也会溢出，所以 `carry = 1`

所以 golden vectors 给出的结果是：

- `sum_out[23:0] = 12, 252, 1246, 97294, 1878286, 169214`
- `carry = 0, 0, 0, 0, 0, 1`

一句话记忆：

`mode 1` 才是“全累加”。

### 2.3 mode 切换到底怎么理解

修订后的 `spec.md` 已经把这件事讲得很明确了，建议直接按下面这套理解：

1. 切换点发生在 `(14,71)` 这一组输入完成之后。
2. 也就是说，前 3 组数据在切换发生前已经被模块接收了。
3. 这前 3 组数据必须继续按旧模式，也就是 `mode 0` 的规则处理和输出。
4. 从第 4 组开始的新输入，允许在旧结果还没完全输出完的时候继续进入模块。
5. 但是这些新输入只能先被接收、先被缓存，不能提前按 `mode 1` 计算并输出。
6. 必须等旧模式待输出结果全部输出完成，而且相关寄存器清零后，后面的新数据才能按新模式 `mode 1` 开始处理和输出。

这句话非常关键：

**允许“先接收”，不允许“先按新模式输出”。**

所以测试组合 3 的正确理解是：

- 前 3 组按旧 `mode 0` 算
- 后 3 组等旧模式结果彻底收尾后，再按新 `mode 1` 重新开始算

于是切换场景的输出应该是：

- 前 3 组：`12, 252, 1234`
- 后 3 组重新按 `mode 1` 累加：
  - 第 4 组：`96048`
  - 第 5 组：`96048 + 1780992 = 1877040`
  - 第 6 组：`1877040 + 15068144 = 16945184`

第 6 组同样超过 24 bit，所以只取低 24 bit：

- `16945184[23:0] = 167968`
- `carry = 1`

因此 golden vectors 给出的切换结果是：

- `sum_out[23:0] = 12, 252, 1234, 96048, 1877040, 167968`
- `carry = 0, 0, 0, 0, 0, 1`

## 3. 这个工程里的文件应该怎么分工

当前这个小工程的目录不复杂，每个文件的职责最好一开始就分清楚：

- `AGENTS.md`：项目规则入口，告诉后续实现时要遵守哪些硬要求。
- `docs/spec.md`：题目规格，尤其要看清修订后的 `mode` 切换解释。
- `docs/golden_vectors.md`：标准答案，用来核对 RTL 和 testbench 的结果是否正确。
- `docs/notes.md`：也就是这份笔记，作用是把题目翻译成更容易理解、更容易实现的话。
- `rtl/mac16.sv`：DUT，也就是真正要实现的硬件模块。
- `tb/tb_mac16.sv`：自检 testbench，负责喂输入、收输出、做比较、打印最终结果。
- `sim/`：仿真过程里放日志、波形或者编译产物的目录。

为什么先写 `notes.md` 很有用？

因为这个题最容易错的不是语法，而是语义。特别是 `mode 0`、`mode 1` 和 `mode` 切换时“旧数据先收尾、新数据可先缓存但不能先输出”这几个规则，只要理解错一个，RTL 和 testbench 往往会一起错。

## 4. 最容易踩的坑

下面这些坑，几乎都很常见：

- 把 `MSB first` 写反成 `LSB first`
- `in_ready` 拉高 16 个周期时，位计数少 1 或多 1
- `inA` 和 `inB` 的 16 bit 没有正确拼成完整无符号数
- 把 `mode 0` 错写成“全累加”
- 把 `mode 1` 错写成“当前乘积 + 上一组乘积”
- 忘记所有数据都是无符号数，结果按有符号数去理解
- 24 bit 截断做错，或者 `carry` 判断条件做错
- 把 `carry` 做成单拍脉冲，而不是保持到 `rst_n` 或 `mode` 切换
- `mode` 一切换就马上清空旧状态，导致旧模式已经接收的数据被丢掉
- 切换后虽然允许继续收新输入，但错误地让这些新输入提前按新模式计算或输出
- `out_ready` 的有效窗口和 `sum_out` 的 24 个输出位没有对齐
- 空闲时 `sum_out` 没有保持为 `0`
- 以为输入结束到输出开始之间必须是固定延迟，但修订版 `spec.md` 已明确说 latency 可以变化

## 5. 推荐的 RTL 架构（简单说法）

如果从“容易做对、也容易检查”的角度出发，一个比较朴素的思路如下。

### 5.1 输入接收器

先做一个输入接收器：

- 当 `in_ready = 1` 时，每个时钟接收 `inA` 和 `inB` 的 1 个 bit
- 连续收满 16 个时钟后，拼成一组完整的 `A` 和 `B`

因为输入是 `MSB first`，所以移位方向一定要和位序对应好。

### 5.2 乘法器

每收满一组 16 bit 数据，就做一次：

`product = A × B`

由于 `16 bit × 16 bit`，所以乘积建议按 32 bit 保存，后面再决定输出如何截成 24 bit。

### 5.3 模式状态寄存器

不同 `mode` 需要记住的历史信息不同：

- `mode 0` 需要记住“上一组乘积”
- `mode 1` 需要记住“当前累加和”

所以内部最好有明确的状态寄存器，而不是把所有规则混在一个大表达式里。

### 5.4 输入缓冲或等待区

这是修订版 `spec.md` 新增口径后最不能省的一块。

因为题目明确说了：

- 不能依赖长时间暂停输入来完成切换
- 旧模式结果没吐完时，新输入仍然可能继续到来

所以切换后必须有一个“先接收、先缓存”的地方。否则一边还在收尾旧模式，一边新输入又进来，就很容易丢数。

### 5.5 输出队列

当一组结果已经算好后，不建议立刻边算边吐。更稳妥的做法是：

- 先把“结果低 24 bit”
- 再把“这一组对应的 `carry`”
- 以及它的输出顺序

一起放进一个输出队列或等待寄存器里。

这样输出部分就只负责按顺序发，不会把“计算逻辑”和“串行发送逻辑”搅在一起。

### 5.6 输出移位器

输出移位器的工作比较直接：

- 从队列头部拿到一组 24 bit 结果
- 按 `MSB first` 一位一位送到 `sum_out`
- 同时在这 24 个输出周期里拉高 `out_ready`
- 发送结束后，如果没有下一组数据，就让 `sum_out` 回到 `0`

### 5.7 mode 切换时推荐的双阶段思路

为了满足修订版 `spec.md` 的要求，推荐把切换理解成两个阶段：

1. **旧模式收尾阶段**
   - 负责把切换前已经接收的数据继续按旧模式处理完
   - 负责把这些旧结果完整输出完
   - 在旧结果还没结束前，不能把它们清掉

2. **新模式等待阶段**
   - 负责接收切换后的新输入
   - 这些新输入先缓存起来
   - 但在旧模式完全结束前，不能提前按新模式输出

等旧模式结果全部输出完成后：

- 清零与旧模式相关的寄存器
- 再把缓存的新输入转入新模式处理

为什么推荐这种思路？

因为它最容易同时满足下面 3 条：

- 旧数据不能丢
- 旧结果必须先完整输出
- 新输入又允许提前进入

对于初学者来说，这种分阶段理解通常比“所有事情在一个状态机里同时硬拧”更不容易出错。
