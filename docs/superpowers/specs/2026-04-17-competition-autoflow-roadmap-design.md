# 比赛题自动化总蓝图设计

## 目标

围绕 [docs/spec.md](E:/GitHub/v0.1_test_workflow/docs/spec.md) 的比赛题目，逐步建设一套可持续迭代的自动化系统，使桌面 Codex、GitHub 侧 GPT-5.4、本地执行环境和远端 EDA 服务器形成闭环，最终尽可能自动完成：

- 需求理解与澄清
- 执行计划生成
- RTL / testbench 修改
- 前端验证
- 远端综合 / STA / 面积 / 功耗等后端流程
- 基于结果的持续优化迭代

这份文档是**总蓝图与路线图**，不是某一阶段的具体实施计划。

---

## 设计原则

### 1. 桌面对话是唯一用户入口

用户应只在 Windows 桌面 Codex 对话里完成日常协作，不要求用户主动操作 GitHub PR、GitHub 网页、workflow 页面或远端服务器。

用户理想上的日常动作只保留三类：

- 提需求
- 回答澄清问题
- 回复极短指令；常见例子包括：
  - `批准`
  - `继续修`
  - `继续后端`
  - `停止优化`
  - `继续深挖`
  - `接受当前最优`
  - `合并`

### 2. GitHub 是后台状态机，不是主要交互面

GitHub PR、评论、标签、workflow、run artifact 仍然是系统的正式持久化与审计面，但这些机制主要服务于：

- 状态流转
- 历史追踪
- 多执行体之间的交接
- 自动化触发

它们不应要求用户亲自频繁进入和操作。

### 3. 桌面 Codex 负责“对用户友好”的桥接

桌面 Codex 不只是代码执行器，也应承担编排代理职责：

- 把用户的自然语言需求转成 request
- 代用户创建 PR、查看状态、发送评论命令
- 把 GitHub 侧 GPT-5.4 的问题转述回桌面对话
- 把用户在桌面对话里的回答，再同步回 PR 持久化
- 在本地与远端执行之间做统一协调

换句话说，用户面对的是桌面 Codex；GitHub 和服务器都在桌面 Codex 背后。

### 4. 先分期闭环，再逐步外扩

这套系统不是“一次性全做完的 workflow”，而是一个有状态的编排系统。必须按阶段逐步落地：

- 先让前端 phase-1 真正稳定可用
- 再接远端后端执行
- 再接自动优化迭代

不要一开始就把所有远端执行、指标分析、自动优化、失败恢复同时压进一个版本。

### 5. Phase-2 必须沿用 Phase-1 的操作风格

phase-2 不应变成另一套新的用户玩法，而应直接继承 phase-1 已经跑通并且用户已经习惯的交互方式。

这意味着：

- 仍然由桌面 Codex 作为唯一用户入口
- 仍然由桌面 Codex 代用户操作 PR、评论、状态和命令
- 仍然使用同一个 request PR 贯穿整条链路
- 仍然使用“桌面对话里一句短指令拍板”的方式推进
- 仍然由 GitHub 负责后台持久化与审计，而不是要求用户直接去 GitHub 操作

## 当前已完成的部分

截至当前仓库状态，已经真实跑通的是 **phase-1 PR-driven frontend autoflow**。

已完成能力包括：

- 用户在桌面 Codex 提需求
- Codex 创建 request 文档并开 GitHub PR
- GitHub 侧 GPT-5.4 做需求澄清与执行计划生成
- 用户批准后，桌面 Codex 本地执行计划
- 本地运行前端验证：
  - RTL 编译
  - testbench 自检
  - golden vector 检查
- 本地生成 run 记录、tag，并推回 GitHub
- GitHub 侧 GPT-5.4 再做 frontend review
- 如有 blocking 问题，支持一轮最小 `/codex-fix` 修复闭环
- 在当前仓库的 standalone phase-1 用法里，通过后可以合并 PR

这部分已经不是概念设计，而是仓库当前主线能力。

## 总体系统分层

建议把整套系统理解为两层：

### A. 交互编排层

负责“谁和谁说话、状态怎么推进”：

- 桌面 Codex 对话
- GitHub PR
- GitHub Actions / workflow
- GPT-5.4 planner / reviewer

### B. 执行计算层

负责“实际跑什么”：

- 本地 Windows 执行环境
- 本地仿真与前端验证工具
- 远端 Linux / EDA 服务器
- 后端流程输出与日志归档

交互编排层负责决定下一步该做什么；执行计算层负责把那一步真正跑出来。

## 角色定义

### 用户

职责：

- 在桌面对话里提出目标
- 在桌面对话里回答问题
- 在桌面对话里做少量拍板

非职责：

- 不要求手动维护 PR 评论
- 不要求自己判断 workflow 卡点
- 不要求自己搬运服务器日志

### 桌面 Codex

职责：

- 用户唯一日常入口
- request / PR 创建者
- GitHub 与桌面对话之间的双向桥接者
- 本地执行器
- 远端服务器流程的操作者
- 结果归档与回传的协调者

### GitHub 侧 GPT-5.4

职责：

- 理解需求
- 发现阻塞性歧义
- 输出执行计划
- 审核前端结果
- 后续阶段审核后端结果
- 必要时输出优化方向

非职责：

- 不直接改代码
- 不直接跑本地或远端工具
- 不直接与桌面线程通信

### 远端 EDA 执行环境

职责：

- 跑编译、综合、时序分析、面积、功耗等后端流程
- 产出 log、report、netlist、约束快照等正式结果

## 统一交互模型

这是整套系统最关键的统一约定。

### 1. 用户永远只和桌面 Codex 对话

即使 GitHub 侧 GPT-5.4 有问题，也不是让用户自己去 PR 回答，而是：

1. GitHub 侧 GPT-5.4 把问题写到 PR
2. 桌面 Codex 读取并转述回当前桌面对话
3. 用户直接在桌面对话里回答
4. 桌面 Codex 再把答案整理后同步回 PR

因此，对用户来说，体验上仍然是“系统在桌面问我，我在桌面答”。

### 2. GitHub PR 保留正式记录

虽然用户不直接操作 PR，但 PR 仍需保存正式记录，因为它是：

- planner / reviewer 的可读上下文
- workflow 触发源
- 状态与审核结论的审计面
- 多阶段工件的统一归档锚点

只要 request 已经进入 phase-2，同一个 request PR 就必须继续保持 open，直到进入后端终态并由用户做最终处理；不能在 `wf:frontend-passed` 时按 phase-1 旧习惯直接提前合并。

### 3. 桌面 Codex 负责命令翻译

系统内部仍可能使用：

- `/answer ...`
- `/approve-plan`
- `/codex-fix`

但这些命令应尽量由桌面 Codex 代发，而不是要求用户直接输入到 PR。

## 三期路线图

## Phase-1：前端需求到前端通过

### 目标

围绕比赛题目前端实现，完成：

- 需求澄清
- 执行计划
- RTL / testbench 修改
- 本地前端验证
- GPT 前端再审

### 当前状态

已基本完成，并已真实跑通。

### 输入

- 用户在桌面对话里的需求
- 仓库中的 `docs/spec.md`
- 仓库中已有 RTL / testbench / golden check 基础

### 输出

- request PR
- plan comment
- 本地 run 记录
- Git tag
- frontend review 结论

### 结束条件

- 在当前 standalone phase-1 用法里：
  - `wf:frontend-passed`
  - 或 `wf:failed`

但在启用 phase-2 的总系统里，`wf:frontend-passed` 只是后端交接点，不是 request PR 的自动合并点。

### 用户体验要求

- 用户只在桌面对话里提需求、答疑、批准、继续修、合并
- 不要求用户自己操作 PR 评论

## Phase-2：远端后端分层推进

### 总体思路

数字后端不能再笼统写成一个模糊的“backend flow”。按照比赛题面与实际依赖关系，应该明确拆成三层：

- **phase-2A：综合 + Formal EC + 基线报告采集**
- **phase-2B：P&R + STA + 功耗/面积主优化循环**
- **phase-2C：LVS + SPEF + 物理验证签核**

三层之间是递进关系，不应在第一版实现里揉成一个自动大闭环。

### 共同要求

1. **服务器结果必须回收到 Windows**
   因为远端服务器可能重启、临时目录可能丢失，所以不能把正式结果只留在远端。

2. **结果必须能回溯到具体版本**
   每次后端运行都必须能对应到：
   - request
   - commit
   - tag
   - 运行时间
   - 运行状态

3. **回传内容要足够 GPT 分析**
   不要求把服务器所有垃圾文件都带回，但必须保留足够用于判断题目达标情况的关键 log / report / summary。

4. **本阶段必须沿用 phase-1 的用户操作风格**
   用户仍然只在桌面对话里回复极短指令；桌面 Codex 负责把这些短指令翻译成 PR 与执行动作。

5. **每一层的启动都先尊重人工确认点**
   至少在从 phase-1 进入 phase-2A 时，必须先停住；只有在用户回复 `继续后端` 之后才可真正启动远端后端。

### Phase-2A：综合 + Formal EC + 基线报告采集

#### 目标

先证明这版 RTL 能稳定进入后端工具链，并完成：

- 逻辑综合
- mapped netlist 生成
- 约束加载确认
- Formal EC
- synthesis power / area / timing summary 采集
- 关键结果拉回 Windows
- 必要工件同步回 GitHub

#### 关键定位

- 这是**基线门**，不是主优化循环。
- 这一层要回答的是：
  - RTL 能不能被综合
  - 综合前后功能是否一致
  - 后端基线报告是否完整可追踪
- synthesis power / area / timing summary 在这一层必须采集，但**不在这一层做多轮自动优化**。

#### 当前状态

这是当前批准的下一步实现范围。

### Phase-2B：P&R + STA + 功耗/面积主优化循环

#### 目标

在 phase-2A 打通基线后，再进入：

- 自动布局布线
- 3 个 PVT corner 的 STA
- 面积 / 功耗 / timing 主循环优化

#### 关键定位

- 这一层才是比赛最终指标的主战场。
- `1GHz`、三组 corner、功耗和面积，主要应在这里进入受控自动优化循环。

#### 当前状态

尚未实现，且当前不进入实施计划。

### Phase-2C：LVS + SPEF + 物理验证签核

#### 目标

在 phase-2B 有较稳定候选版本后，再进入：

- LVS
- SPEF
- 物理验证
- 更接近最终签核的收口检查

#### 关键定位

- 这一层更像阶段签核或最终交付门，而不是每一轮都要跑的主循环。

#### 当前状态

尚未实现，且当前不进入实施计划。

## Phase-3：深挖优化与最终收口

### 目标

在 phase-2B 的默认自动循环之外，phase-3 负责更大策略切换和最终收口，例如：

- `backend-stalled` 后的“继续深挖”模式
- 更大级别的结构调整
- 默认 phase-2B 优化策略之外的专项优化
- 赛事最终交付前的人工收口与最终接受

phase-3 不再表示“普通 backend optimize 每轮都要走的默认路径”，而表示默认 phase-2B 自动循环之外的更高阶阶段。

### 本阶段的难点

- 优化目标往往是多指标的：
  - 正确性
  - 频率
  - 面积
  - 功耗
- 各指标之间可能互相冲突
- 自动优化必须避免无止境震荡

### 当前状态

尚未实现。

### 本阶段原则

在真正实现前，不应假设深挖优化总能自动收敛；必须保留人工停止和人工接管能力。

## 版本与 tag 原则

系统需要持续使用**带时间戳和运行状态的版本标签**，以便后续前端、后端、优化三种阶段统一追踪。

建议继续沿用当前已经跑通的 ASCII 友好格式，而不是自由格式中文 tag。推荐格式：

- `v0.1_frontend_pass_YYYYMMDD_HHMMSS_rNNN`
- `v0.1_frontend_rework_YYYYMMDD_HHMMSS_rNNN`
- `v0.1_backend_pass_YYYYMMDD_HHMMSS_rNNN`
- `v0.1_backend_fail_YYYYMMDD_HHMMSS_rNNN`
- `v0.1_opt_iter_YYYYMMDD_HHMMSS_rNNN`

原因：

- 便于脚本解析
- 便于跨平台工具处理
- 便于在 GitHub、Windows、本地脚本、远端脚本之间统一引用

用户需要看到“运行状态”和“时间”，但不需要亲自维护 tag 规则。

## 成功标准

这套总系统最终成功，不是指“某一条 workflow 能运行”，而是指：

1. 用户只通过桌面对话就能驱动整套流程
2. GitHub 作为后台状态面稳定记录所有关键状态与产物
3. 前端修改、后端运行、结果归档可以版本化追踪
4. 服务器重启不会导致关键结果丢失
5. GPT-5.4 能基于前端或后端结果给出下一步明确判断
6. 系统能够持续逼近 `docs/spec.md` 的比赛目标，而不是停在“能跑 workflow”
